# 09. Hooks 自动化

Hooks 用于在 Claude Code 的特定事件前后自动执行脚本。它适合把团队规则固化为工具行为，例如格式检查、命令拦截、日志记录或提醒。

## Hooks 适合解决什么问题

- 每次编辑后自动格式化
- 运行命令前检查是否危险
- 会话结束时发送通知
- 阻止提交敏感文件
- 自动记录操作日志
- 根据项目策略限制某些命令

## Hooks 与普通提示词的区别

提示词依赖模型遵守，hooks 由 Claude Code harness 执行，更适合强制性自动化。

例如：

- “以后每次提交前都检查密钥”更适合 hook
- “这次提交前帮我检查密钥”可以用普通提示词

## 配置位置

Hooks 通常配置在 Claude Code 的 settings.json 或项目配置中。具体路径和字段以当前版本为准。

常见层级：

- 用户级设置
- 项目级设置
- 本地私有设置

## Hook 设计原则

### 1. 快速

Hook 会打断交互流程，不能太慢。

### 2. 可解释

失败时输出清晰原因和修复建议。

### 3. 最小副作用

不要在 hook 中做危险操作，例如删除文件、部署或修改远程资源。

### 4. 可重复运行

Hook 应该幂等，多次运行不应造成状态混乱。

## 适合做 Hook 的检查

- 阻止提交 `.env`
- 阻止运行 `rm -rf /`
- 检查 package manager 是否一致
- 检查是否在正确目录运行命令
- 提醒先运行测试
- 自动格式化暂存文件

## 不适合做 Hook 的事情

- 长时间全量测试
- 生产部署
- 自动 force push
- 自动删除分支
- 依赖不稳定外部服务的阻塞检查

## 示例：危险命令拦截思路

当 Claude Code 准备运行 shell 命令时，hook 可以检查命令字符串。如果包含危险模式，就拒绝并提示用户手动确认。

需要拦截的模式可能包括：

- `rm -rf`
- `git reset --hard`
- `git clean -fd`
- `git push --force`
- `drop database`

## 示例：会话结束通知

在长任务完成后，可以用 hook 发送桌面通知或写日志。但通知应克制，避免每次小任务都打扰用户。

## 调试 Hooks

如果 hook 阻止了操作：

1. 阅读 hook 输出
2. 判断是规则正确拦截还是误报
3. 修改命令或调整 hook
4. 不要简单绕过安全检查

## 配置建议

团队可以把稳定规则放进项目配置，把个人偏好放进用户配置。

项目适合配置：

- 禁止提交密钥
- 常用只读命令允许列表
- 关键目录保护

个人适合配置：

- 通知方式
- 编辑器偏好
- 本地工具路径

## 让 Claude Code 帮你配置 Hooks

可以这样说：

```text
帮我配置一个 hook：当我要运行 git push --force 时必须阻止并提示我手动确认。
```

如果要修改 settings.json，建议明确配置范围：用户级、项目级或本地级。

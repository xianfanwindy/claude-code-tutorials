# 16. 安全审查与防御性使用

Claude Code 可以辅助安全审查、防御性测试、CTF 和授权的安全研究。关键是明确授权边界和避免破坏性行为。

## 适合 Claude Code 的安全任务

- 审查当前 diff 是否引入漏洞
- 检查鉴权和权限边界
- 查找硬编码密钥
- 分析依赖风险
- 编写安全回归测试
- 改进输入校验
- 修复 XSS、SQL 注入等漏洞
- 为内部安全流程写文档

## 不适合或必须拒绝的任务

- 未授权攻击
- 大规模扫描公共目标
- 凭据窃取
- 绕过检测用于恶意目的
- 破坏性 DoS
- 供应链投毒
- 删除或篡改他人数据

## 安全审查范围

审查代码时，可以关注：

### 输入处理

- 是否验证用户输入
- 是否存在命令注入
- 是否存在 SQL 注入
- 是否存在路径穿越
- 是否正确处理文件上传

### 输出处理

- 是否有 XSS
- 是否泄漏敏感字段
- 错误信息是否过度暴露

### 鉴权与授权

- 是否检查登录状态
- 是否检查资源所有权
- 管理员接口是否受保护
- 多租户隔离是否正确

### 密钥和配置

- 是否硬编码 token
- 日志是否打印密钥
- 默认配置是否安全

### 依赖和供应链

- 是否引入不必要依赖
- 依赖是否可信
- install script 是否危险

## 审查提示词

```text
对当前 diff 做安全审查，重点关注 OWASP Top 10。只报告高可信问题，不要修改代码。
```

```text
检查这个 API endpoint 是否存在越权访问风险，并说明攻击路径和修复建议。
```

```text
查找仓库中可能被提交的密钥或敏感配置。
```

## 修复安全问题

修复时应：

1. 写出漏洞成因
2. 最小化修改
3. 添加回归测试
4. 避免只做表面过滤
5. 运行安全相关验证

## 安全测试说明授权

如果进行安全测试，应明确：

```text
这是授权测试，目标仅限本地开发环境，不允许访问外部目标，不允许 DoS，不允许读取真实用户数据。
```

## 生成 PoC 的边界

在防御性上下文中，可以生成最小复现用于验证修复。但应避免提供可直接用于攻击第三方的完整武器化流程。

## 安全 Review 输出模板

```markdown
## Findings

### High

- ...

### Medium

- ...

### Low

- ...

## Tests Recommended

- ...

## Notes

- 未发现并不等于没有漏洞。
```

## 团队建议

- 对认证、支付、权限、文件上传等模块强制安全审查
- 对依赖新增进行额外 review
- 对日志和错误处理检查敏感信息
- 对生产配置单独审查
- 保留安全修复的测试用例

# 08. 权限、安全边界与审批

Claude Code 能执行真实操作，因此权限管理非常重要。安全的目标不是阻止效率，而是让高风险操作可见、可控、可回滚。

## 权限模型的意义

Claude Code 调用工具时，某些操作会请求用户批准，例如：

- 运行 shell 命令
- 编辑文件
- 创建文件
- 推送代码
- 调用外部服务
- 修改配置

用户可以批准、拒绝或调整后继续。

## 风险分级

### 低风险

- 读取文件
- 查看 Git 状态
- 搜索代码
- 运行只读命令

### 中风险

- 编辑项目文件
- 运行测试
- 安装依赖
- 生成提交

### 高风险

- 删除文件
- 修改数据库
- force push
- 部署生产环境
- 修改权限
- 发送外部消息
- 上传敏感内容

## 危险操作需要确认

以下操作应要求明确确认：

- `rm -rf`
- `git reset --hard`
- `git clean -fd`
- `git push --force`
- 删除分支
- 删除数据库表
- 生产部署
- 修改云资源
- 发送邮件或 Slack 消息

## 最小权限原则

只授权当前任务需要的能力。

例如，只需要跑测试时，不要授权部署命令；只需要读 GitHub issue 时，不要授予写权限 token。

## 环境变量和密钥

安全规则：

- 不打印完整密钥
- 不把密钥写入日志
- 不提交 `.env`
- 不上传凭据到第三方工具
- 使用专用 token 而非个人主 token
- 定期轮换

好指令：

```text
检查 GITHUB_TOKEN 是否存在，但不要输出它的值。
```

## 安全测试边界

Claude Code 可以协助授权的安全测试、防御性审查、CTF 和教育场景。

不应请求：

- 未授权攻击
- 破坏性操作
- 大规模扫描
- 凭据窃取
- 绕过检测以逃避追踪
- 供应链投毒

如果是合法测试，应说明：

- 授权范围
- 目标系统
- 禁止操作
- 测试窗口
- 输出要求

## 代码安全重点

让 Claude Code 修改代码时，可要求关注：

- SQL 注入
- 命令注入
- XSS
- SSRF
- CSRF
- 鉴权绕过
- 越权访问
- 敏感信息泄漏
- 不安全反序列化
- 路径穿越

提示词：

```text
审查当前 diff 是否引入 OWASP Top 10 风险。
```

## 配置权限白名单

对于频繁使用的只读命令，可以配置允许列表，减少重复提示。但不要把危险命令加入白名单。

适合允许：

- `git status`
- `git diff`
- `git log`
- 测试命令
- 类型检查命令

不适合默认允许：

- 删除命令
- 部署命令
- force push
- 修改云资源命令

## 审批沟通模板

当 Claude Code 请求高风险操作时，它应该说明：

```text
我准备执行 X。
原因：Y。
影响范围：Z。
可逆性：可以/不容易。
是否继续？
```

## 团队安全建议

- 明确哪些命令可自动批准
- 使用项目级 settings 管理权限
- 对生产操作要求人工确认
- 对安全相关改动强制 review
- 对 AI 生成代码运行测试和扫描
- 不允许 AI 自动处理真实密钥

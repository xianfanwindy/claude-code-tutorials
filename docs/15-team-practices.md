# 15. 团队协作最佳实践

团队使用 Claude Code 时，核心目标是让 AI 协作变得一致、安全、可审查，而不是每个人各用各的临时方式。

## 团队落地清单

建议统一：

- CLAUDE.md
- 常用命令
- 权限策略
- hooks 配置
- PR 模板
- 安全审查要求
- 不允许 AI 执行的操作
- 推荐提示词模板

## 项目级 CLAUDE.md

团队应维护项目级 CLAUDE.md，记录：

- 技术栈
- 构建和测试命令
- 目录职责
- 代码风格
- Review 要求
- 安全边界
- 发布流程注意事项

## 权限策略

可按风险分层：

### 默认允许

- 读取文件
- 搜索代码
- 查看 git status / diff
- 运行项目测试

### 需要确认

- 编辑文件
- 安装依赖
- 创建提交
- 创建 PR

### 严格限制

- force push
- 删除数据
- 生产部署
- 修改权限
- 发送外部消息

## PR 规范

建议要求 AI 生成 PR 时包含：

```markdown
## Summary

- ...

## Test plan

- [x] ...
- [ ] ...

## Risk

- ...
```

## Code review 规范

AI 可以帮助 review，但不能替代人类责任。建议：

- AI 先做机械和安全检查
- 人类 review 业务语义和产品意图
- 高风险模块必须有人类 owner 审查
- AI review 结果需要人工判断

## 安全要求

团队应明确：

- 不处理生产密钥
- 不自动执行生产写操作
- 不绕过 CI 和 hooks
- 不把内部代码上传到未批准服务
- 对安全相关改动执行 security review

## 提示词模板库

团队可以维护常用提示词：

### Bug 修复

```text
请先复现/定位问题，不要修改代码。找到根因后给出最小修复方案和验证方式。
```

### 新功能

```text
先阅读相关模块并制定计划，说明要改哪些文件、风险和测试策略，等我确认后再实现。
```

### Review

```text
审查当前 diff，重点关注正确性、边界条件、安全问题和测试缺口。
```

### PR

```text
创建 PR，标题不超过 70 字符，描述包含 Summary、Test plan 和 Risk。
```

## 培训建议

新成员应学习：

1. 如何限制 Claude Code 只读调查
2. 如何要求最小 diff
3. 如何审查 AI 修改
4. 如何运行验证
5. 哪些操作必须人工确认

## 度量指标

可以观察：

- PR 周期是否缩短
- 测试覆盖是否改善
- Review 缺陷是否减少
- 事故或误操作是否增加
- hooks 是否频繁误报

## 反模式

- 让 AI 自动合并 PR
- 让 AI 默认部署生产
- 不看 diff 直接提交
- 把所有权限一次性放开
- 用 AI 生成大量无测试代码
- 用 memory 保存临时状态或密钥

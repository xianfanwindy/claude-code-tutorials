# 14. 计划模式、工作树与长任务

Claude Code 支持多种高级工作流，适合复杂实现、并行实验和长时间任务。

## 计划模式

当任务复杂、影响多文件或存在多种实现方案时，建议先进入计划模式。

适合场景：

- 添加认证系统
- 重构核心模块
- 改造错误处理
- 添加缓存策略
- 优化性能
- 多文件功能开发

计划模式流程：

1. 探索代码
2. 理解现有模式
3. 设计实现方案
4. 写出计划
5. 用户批准
6. 再开始实现

提示词：

```text
先制定实现计划，不要修改代码。说明会改哪些文件、风险和验证方式。
```

## Worktree 工作流

Git worktree 允许在同一仓库创建隔离工作目录。适合：

- 实验性修改
- 并行开发多个分支
- 避免污染当前工作区
- 长任务隔离

使用前应确认：

- 当前仓库干净或清楚未提交改动
- 新 worktree 的基准分支
- 是否需要保留 worktree

不要把 worktree 当成随意删除改动的工具。

## 长任务

长任务包括：

- 全量测试
- 构建
- 大型搜索
- CI 轮询
- 部署状态观察

Claude Code 可以后台运行命令或设置监控。

## 后台任务

适合：

```text
启动测试，完成后告诉我结果。
```

对于长命令，应避免频繁轮询。系统通常会在任务完成后通知。

## Monitor

Monitor 适合持续监听事件，例如：

- 日志中出现 ERROR
- CI 状态变化
- 文件变化
- 服务启动成功

关键原则：

- 只输出有用事件
- 覆盖失败状态
- 不要只监听成功关键字
- 避免产生大量通知

## Loop

Loop 适合重复执行任务：

```text
/loop 5m 检查这个 PR 的 CI 状态，如果失败就总结原因
```

适合：

- 定期检查 CI
- 轮询部署
- 定期审查新评论

不适合：

- 一次性任务
- 本地后台命令完成等待

## 任务清单

复杂任务可以让 Claude Code 建立任务列表：

```text
把这个迁移任务拆成 todo，并按顺序执行。
```

任务清单适合当前会话，不适合作为长期记忆。

## 组合工作流示例

```text
我们要给 API 添加请求级限流。
请先进入计划模式：
1. 调查现有中间件；
2. 比较可行方案；
3. 给出最小实现计划；
4. 等我批准后再改代码；
5. 修改后运行相关测试。
```

## 风险控制

高级工作流常涉及更大影响面，因此要：

- 明确授权范围
- 控制并行修改
- 避免自动执行破坏性操作
- 保留可 review 的 diff
- 在关键节点让用户确认

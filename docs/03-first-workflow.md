# 03. 第一个项目工作流

本章用一个典型 bug 修复流程说明如何和 Claude Code 协作。

## 工作流总览

推荐流程：

1. 明确目标
2. 让 Claude Code 只读探索
3. 确认修改方案
4. 小步编辑文件
5. 运行测试或手动验证
6. 查看 diff
7. 提交代码
8. 创建 PR 或推送

## 第一步：明确目标

不要只说：

```text
修一下 bug
```

更好的方式：

```text
登录后刷新页面会回到未登录状态。请先调查原因，不要修改代码，找到相关文件和可能的根因。
```

明确的信息越多，Claude Code 越容易做对：

- 现象
- 复现步骤
- 期望行为
- 实际行为
- 是否允许修改
- 是否需要测试

## 第二步：只读探索

让 Claude Code 先阅读代码：

```text
先定位登录状态持久化的实现，列出相关文件和调用链，不要改文件。
```

这样可以避免它过早修改错误位置。

## 第三步：确认方案

对于非平凡改动，可以要求：

```text
给我一个简短修改方案，说明会改哪些文件、为什么这么改，以及怎么验证。
```

如果任务比较复杂，Claude Code 可能进入计划模式，先产出计划，再等你批准。

## 第四步：执行修改

确认方案后：

```text
按这个方案修改，保持改动尽量小。
```

建议强调：

- 不要顺手重构无关代码
- 不要改格式化之外的无关文件
- 不要引入新依赖，除非必要
- 保持变更可 review

## 第五步：运行测试

让 Claude Code 根据项目识别测试命令：

```text
运行相关测试。如果没有精确测试，运行最小必要的验证命令。
```

对于前端 UI，还应要求真实运行：

```text
启动应用并在浏览器里验证这个交互，不要只跑类型检查。
```

## 第六步：查看 diff

提交前让 Claude Code 总结 diff：

```text
总结当前 diff，指出改了哪些行为、有哪些风险、跑了哪些验证。
```

也可以要求它自查：

```text
审查你刚才的改动，重点找逻辑错误、边界条件和安全问题。
```

## 第七步：提交

如果你希望提交：

```text
为当前改动创建一个 git commit，不要 push。
```

如果希望推送：

```text
提交并推送到当前分支。
```

推送会影响远程仓库，通常需要明确授权。

## 第八步：创建 PR

```text
创建一个 PR，标题简短，描述包含 Summary 和 Test plan。
```

Claude Code 会查看当前分支、diff、提交历史，并用 GitHub 工具创建 PR。

## 一个完整示例

```text
用户资料页点击保存后按钮一直 loading。请：
1. 先调查原因，不要修改代码；
2. 找到根因后给出修改方案；
3. 经我确认后再修改；
4. 运行相关测试；
5. 总结 diff 和风险。
```

## 常见误区

### 一开始就要求大改

不推荐：

```text
重写整个认证模块
```

推荐：

```text
先评估认证模块有哪些问题，列出可以分阶段改进的方案。
```

### 没有验证要求

不推荐：

```text
改好了告诉我
```

推荐：

```text
改完后运行相关测试，并说明还需要我手动验证什么。
```

### 没说明是否允许推送

推送、发 PR、部署、改远程配置都属于影响共享状态的操作，应明确授权。

# 06. 文件编辑、测试与验证

Claude Code 可以读取和编辑文件，也可以运行测试命令。但高质量结果依赖清晰的修改边界和验证要求。

## 文件编辑原则

### 优先编辑已有文件

对于 bug fix 和小功能，优先修改已有文件，不要轻易新增抽象、工具库或文档。

### 保持最小变更

好指令：

```text
只修复这个问题，不做无关重构，保持 diff 最小。
```

### 不要覆盖用户改动

如果仓库已有未提交改动，Claude Code 应先检查 Git 状态，避免覆盖他人的工作。

你可以要求：

```text
先查看 git status，确认有哪些未提交改动，再开始修改。
```

## 测试类型

### 单元测试

适合验证函数、组件、服务逻辑。

```text
运行与本次修改相关的单元测试。
```

### 集成测试

适合验证数据库、API、队列、外部服务边界。

```text
运行相关集成测试，不要用 mock 替代真实数据库，除非项目本来这样设计。
```

### 类型检查

适合 TypeScript、Python 类型工具、Go 编译等。

```text
运行类型检查。
```

### Lint 和格式化

```text
运行 lint。如果只是格式问题，使用项目已有格式化命令修复。
```

### 手动验证

前端、TUI、Electron、CLI 交互等场景，测试不等于真实可用。

```text
启动应用并在浏览器中验证用户能完成保存流程。
```

## 验证优先级

推荐顺序：

1. 最小相关测试
2. 类型检查
3. lint
4. 更大范围测试
5. 手动验证

不要一开始就跑全量耗时测试，除非任务要求或项目规模很小。

## 失败时怎么做

如果测试失败，不要立刻绕过。应该：

1. 读取失败信息
2. 判断是否和本次修改相关
3. 修复根因
4. 重新运行失败测试
5. 总结剩余失败

指令示例：

```text
测试失败后先分析根因，不要跳过测试，不要使用 --no-verify。
```

## UI 验证要求

对于 UI 改动，建议要求：

```text
启动开发服务器，打开页面，验证主路径和一个边界情况。若无法打开浏览器，请明确说明。
```

需要观察：

- 页面是否加载
- 交互是否可用
- 控制台是否报错
- 网络请求是否成功
- loading、空状态、错误状态是否合理

## 验证报告模板

完成后可以让 Claude Code 输出：

```markdown
## 修改内容

- ...

## 验证

- [x] npm test -- user.test.ts
- [x] npm run typecheck
- [ ] 手动验证：无法执行，原因是 ...

## 风险

- ...
```

## 常见问题

### 只跑了 lint，没跑测试

lint 只能证明代码风格，不证明行为正确。

### 只跑了测试，没验证 UI

测试通过不代表浏览器里的交互路径可用。

### 测试命令不存在

让 Claude Code 读取 package.json、README 或 CI 配置，找出真实命令。

### 测试太慢

先运行相关测试，再根据风险决定是否全量运行。

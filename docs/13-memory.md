# 13. Memory 记忆系统

Memory 用于跨会话保存长期有用的信息，例如用户偏好、项目背景、外部参考位置和协作反馈。

## Memory 适合保存什么

### 用户偏好

例如：

- 用户喜欢简短总结
- 用户希望先给计划再实现
- 用户熟悉后端但不熟悉前端

### 协作反馈

例如：

- 不要自动提交
- UI 改动必须手动验证
- 测试不要 mock 数据库

### 项目背景

例如：

- 某个重构是合规要求驱动
- 某个模块近期被冻结
- 某个发布窗口的约束

### 外部参考

例如：

- bug 在哪个 Linear project 跟踪
- 监控 dashboard 地址
- 设计文档所在位置

## 不适合保存什么

- 可以从代码读出的架构细节
- 临时任务状态
- 当前分支 diff
- 已提交的修复方案
- 密钥或敏感信息
- 很快过期的日志摘要

## 为什么不能什么都记

Memory 会影响未来对话。如果保存过多临时信息，会导致：

- 过期信息误导判断
- 上下文变脏
- 隐私风险增加
- 维护成本变高

## 让 Claude Code 记住信息

可以说：

```text
记住：我希望你在改前端功能后总是启动浏览器验证。
```

Claude Code 应把它保存为未来协作偏好。

## 让 Claude Code 忘记信息

```text
忘记我之前关于不要自动运行测试的偏好。
```

## Memory 与 CLAUDE.md 的区别

| 信息类型 | 放哪里 |
| --- | --- |
| 团队项目规范 | CLAUDE.md |
| 个人协作偏好 | Memory |
| 临时任务清单 | 当前任务列表 |
| 代码结构 | 读取代码 |
| 外部系统位置 | Memory 或 CLAUDE.md，视团队共享程度而定 |

## 使用 Memory 的好习惯

- 只保存长期有用的信息
- 保存原因和适用范围
- 如果过期就更新或删除
- 不保存秘密
- 对当前代码状态先验证再相信记忆

## 示例：好的 Memory

```text
用户希望实现前先说明会改哪些文件和为什么。
原因：用户偏好可审查的小步协作。
适用：非平凡代码修改任务。
```

## 示例：不好的 Memory

```text
今天测试失败了。
```

这属于临时状态，不应保存。

## 团队建议

- 项目公共规则写 CLAUDE.md
- 个人偏好写 Memory
- 安全策略尽量写项目配置或 hooks
- 重要外部系统位置可写 Memory，但要定期验证
